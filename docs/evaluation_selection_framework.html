GraphRAG Kubernetes — Thesis Evaluation Methodology
Systematic Selection Reproducible Graph-Aware Thesis-Grade
1Research Context & Selection Rationale

The system being evaluated is a GraphRAG chatbot that answers Kubernetes questions by traversing a Neo4j knowledge graph built from the Kubernetes OpenAPI (Swagger) specification. Its evaluation framework measures three dimensions: AnsQ (40%) — answer quality, RetQ (35%) — retrieval quality against expected graph nodes, and ReaQ (25%) — reasoning quality along expected traversal paths.


This creates a hard constraint: every question selected for evaluation must be answerable using information that exists in the Kubernetes API definition graph. A question about debugging a live cluster, choosing a cloud provider, or using Helm cannot be evaluated by this system's metrics — it wouldn't be a fair test. The selection framework below operationalises this constraint into a reproducible, auditable process.

📌 Why a framework matters: Without explicit criteria, dataset selection introduces confirmation bias — picking questions the system can answer well. A published, pre-registered selection methodology prevents this and makes your evaluation credible for a thesis committee.
2Decision Flow — Step by Step

3Gate Definitions — Full Criteria
G1
Kubernetes API Scope
HARD

The question must be answerable using the Kubernetes OpenAPI spec — not operational knowledge, not cloud provider behavior, not third-party tools.

Pass if: answer involves API resource fields, YAML structure, resource relationships, or API semantics.

Fail examples:

  • How do I install Kubernetes on Ubuntu? (infra)
  • Why is my EKS node NotReady? (cloud/ops)
  • How do I use Helm to deploy X? (tool-specific)
  • kubectl port-forward not working (CLI/ops)
G2
Node Existence in Graph
HARD

The primary Kubernetes resource referenced must exist as a Definition node in the Neo4j graph (ingested from the Swagger spec).

Verify using:

  • MATCH (d:Definition) WHERE d.name CONTAINS 'X' RETURN d.name
  • Resource must be in kubernetes_swagger.json definitions

Fail examples:

  • Questions about CRDs (CustomResourceDefinition) that are user-defined
  • Operator-specific resources (PrometheusRule, ServiceMonitor)
G3
API Stability / Deprecation
SOFT

Prefer questions about stable APIs (GA). Questions about beta/alpha or deprecated API versions reduce evaluation reliability since the swagger is pinned to 1.29.

Pass: apps/v1, core/v1, networking.k8s.io/v1, batch/v1, rbac/v1

Fail:

  • extensions/v1beta1 (removed in 1.22)
  • policy/v1beta1 PodSecurityPolicy (removed in 1.25)
  • Questions referencing K8s < 1.21 behavior
G4
Complementarity
SOFT

New questions should expand the evaluation coverage, not replicate what synthesized fixtures already cover. Compare against the 41 existing fixtures before scoring.

Pass (new coverage): new resource type, new question pattern, new traversal path, edge cases not tested.

Deprioritise (duplicate): another "what is Deployment?" or "create nginx Deployment YAML" — already well-covered. Enter scoring with −0.3 penalty.

45-Dimension Scoring Rubric

After passing all gates, each question receives a score of 1–3 on five dimensions. The weighted total determines final acceptance. Max possible score = 3.0.

Score Formula:

Total = (D1×0.30) + (D2×0.25) + (D3×0.20) + (D4×0.15) + (D5×0.10)

Apply −0.30 penalty if Gate 4 flagged duplicate intent.

Scores ≥ 2.0 → ACCEPT  |  1.8–1.9 → REVIEW (manual decision)  |  <1.8 → DISCARD

6Selection Algorithm (Pseudocode)
# GraphRAG Kubernetes — Evaluation Question Selection Algorithm # Input : stackoverflow-kubernetes-questions (10K–100K rows) # Output: 15–20 selected questions for tests/fixtures/realworld/ FUNCTION select_questions(dataset): candidates = [] coverage_tracker = load_existing_fixture_coverage() # 41 synthesized fixtures quota = {type: 8, resource: 5} # max per bucket FOR row IN dataset: q = row.question_title + " " + row.question_body a = row.answer_body score = row.answer_score # ── HARD GATES ────────────────────────────────────────────── IF NOT is_api_spec_answerable(q, a): CONTINUE # G1 resource = extract_primary_resource(q, a) IF NOT exists_in_neo4j_graph(resource): CONTINUE # G2 # ── SOFT GATES ────────────────────────────────────────────── IF uses_deprecated_api(q, a): SKIP_FLAG = True # G3 penalty = 0.0 IF duplicates_existing_fixture(q, coverage_tracker): penalty = -0.30 # G4 # ── SCORING ───────────────────────────────────────────────── d1 = score_graph_answerability(q, a, resource) # 1–3, weight 0.30 d2 = score_path_annotability(q, a, resource) # 1–3, weight 0.25 d3 = score_question_type_fit(q, a) # 1–3, weight 0.20 d4 = score_so_quality(score, q) # 1–3, weight 0.15 d5 = score_real_world_representativeness(q) # 1–3, weight 0.10 total = (d1*0.30) + (d2*0.25) + (d3*0.20) + (d4*0.15) + (d5*0.10) + penalty IF total >= 2.0: fixture_type = map_to_fixture_type(q) candidates.append({ "question": q, "resource": resource, "fixture_type": fixture_type, "total_score": total, "scores": [d1, d2, d3, d4, d5], "answer": a }) # ── DIVERSITY-AWARE SELECTION ──────────────────────────────── candidates.sort(key=lambda x: x.total_score, reverse=True) selected = [] type_counts = defaultdict(int) resource_counts = defaultdict(int) FOR c IN candidates: IF type_counts[c.fixture_type] >= quota["type"]: CONTINUE IF resource_counts[c.resource] >= quota["resource"]: CONTINUE IF coverage_tracker.is_needed(c.resource, c.fixture_type): selected.append(c) type_counts[c.fixture_type] += 1 resource_counts[c.resource] += 1 IF len(selected) >= 20: BREAK RETURN selected

8Output Specification — Selected Fixture Format

Selected questions are saved to tests/fixtures/realworld/ with two additional fields (selection_score and so_question_id) for full traceability. The selection_rationale field documents the per-question reasoning, making the evaluation methodology auditable by a thesis examiner.
